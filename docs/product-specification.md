## Overview

This repository is a research workbench for learning chaotic gravitational dynamics with graph neural networks. It generates controlled N-body trajectory datasets, trains model families with different inductive biases, evaluates trained checkpoints with reproducible numeric reports, and aggregates the results needed for thesis tables.

The product is designed around one question: given the same simulated system and the same evidence pipeline, how do direct state-transition models compare with Hamiltonian-inspired models when they are rolled forward over time?

The workbench currently supports two model families:

- EGNN: an equivariant graph neural network that predicts the next state directly from the current state.
- HGNN: a Hamiltonian graph neural network that predicts dynamics through a learned Hamiltonian structure.

The program is not a general-purpose physics engine, a model-serving platform, or a user interface application. Its value is in making the thesis experiments repeatable: data comes from explicit configuration, checkpoints carry provenance, official metrics are generated by one evaluator, and scaling-study tables are rendered from machine-readable evaluation files rather than copied from plots.

## Product Schematic

- Data generation creates train, validation, and test HDF5 files from one YAML configuration. The generated state format is fixed: each frame contains positions, velocities, and mass for each particle.
- Dataset loading turns trajectories into one-step supervised transitions. A trajectory with 200 frames provides 199 transitions.
- Training runs one model configuration at a time, writes checkpoints and training metrics, and records enough metadata to evaluate the run later.
- Scaling runs repeat training for several training-set sizes, such as 1000, 2000, and 5000 trajectories.
- Evaluation loads a checkpoint and a test HDF5 file, runs one-step and autoregressive rollouts, measures physical-energy behavior, and writes official JSON and CSV outputs.
- Reporting reads evaluation outputs from multiple runs and renders markdown tables for the data-scaling study.
- Colab operation provides a notebook launcher for long GPU runs, with inputs and outputs anchored in Google Drive.

The intended operating loop is:

1. Generate or provide the dataset files.
2. Train EGNN and HGNN checkpoints under the same data split.
3. Evaluate the best checkpoint for each run.
4. Aggregate results into thesis-ready tables.
5. Use visual diagnostics only after the numeric reports exist.

## Supported Use Cases

The workbench supports the following client-facing workflows:

- Generate a complete N-body dataset from a reproducible configuration.
- Train EGNN or HGNN on a fixed dataset.
- Override the number of training trajectories without editing YAML files.
- Run a data-scaling sweep for one model.
- Run an EGNN learning-rate and noise-injection sweep.
- Evaluate a trained checkpoint on a test dataset.
- Produce official metrics for single-step prediction, rollout stability, divergence thresholds, and energy drift.
- Build a markdown scaling report across models and dataset sizes.
- Run the same training and evaluation flow on Google Colab with Google Drive persistence.

The workbench intentionally treats official numeric evaluation as a product surface. A result should be cited from `metrics.json`, `summary.csv`, or a generated scaling report. Plots and notebooks are diagnostic tools, not the primary source for thesis tables.

## Command Interfaces

All commands are run from the repository root. The preferred local invocation is through `uv`:

```bash
uv run python -m <module> <arguments>
```

Plain `python -m <module>` is also valid when the environment has already been created and dependencies are installed.

### Generate Datasets

```bash
uv run python -m data.generate --config configs/data.yaml
```

Arguments:

| Argument | Required | Default | Description |
| --- | --- | --- | --- |
| `--config` | No | `configs/data.yaml` | Path to the data-generation YAML file. |

What happens:

- Loads the data-generation configuration.
- Simulates independent N-body trajectories.
- Writes train, validation, and test HDF5 files.
- Stores trajectory arrays, physical-energy arrays, and dataset metadata.

Primary outputs:

- `train_path` from the YAML config.
- `val_path` from the YAML config.
- `test_path` from the YAML config.

### Train One Model

```bash
uv run python -m training.train --config configs/egnn.yaml
uv run python -m training.train --config configs/hgnn.yaml
```

For data-scaling runs:

```bash
uv run python -m training.train --config configs/egnn.yaml --n-train 1000
```

Arguments:

| Argument | Required | Default | Description |
| --- | --- | --- | --- |
| `--config` | Yes | None | Path to the model training YAML file. |
| `--n-train` | No | Config value | Overrides `data.n_train_trajectories` for scaling experiments. |

What happens:

- Loads the model, data, training, scheduler, checkpointing, and logging configuration.
- Builds the configured model family.
- Loads one-step transitions from the configured training and validation HDF5 files.
- Trains for the configured number of epochs.
- Writes `latest.pt` every epoch and updates `best.pt` when validation loss improves.
- Writes a training metrics CSV with epoch, train loss, validation loss, and learning rate.

Primary outputs:

- `checkpoints/<model>/<run_id>/latest.pt`
- `checkpoints/<model>/<run_id>/best.pt`
- `logs/<model>/<run_id>/metrics.csv`

### Run a Data-Scaling Sweep

```bash
uv run python -m training.scaling --config configs/egnn.yaml --sizes 1000,2000,5000
```

Arguments:

| Argument | Required | Default | Description |
| --- | --- | --- | --- |
| `--config` | Yes | None | Path to the model training YAML file. |
| `--sizes` | No | `1000,2000,5000` | Comma-separated values for `n_train_trajectories`. |

What happens:

- Loads one base training configuration.
- Repeats training once for each requested training-set size.
- Uses the same model family and hyperparameters across sizes except for the training-trajectory override.

Primary use:

- Produce comparable EGNN or HGNN checkpoints for the thesis data-scaling study.

### Run an EGNN Sweep

```bash
uv run python -m training.sweep --config configs/egnn.yaml --epochs 200
```

Arguments:

| Argument | Required | Default | Description |
| --- | --- | --- | --- |
| `--config` | No | `configs/egnn.yaml` | Base EGNN training configuration. |
| `--epochs` | No | `200` | Number of epochs for each sweep run. |

What happens:

- Runs a small grid over learning rate and `noise_factor`.
- Uses the configured EGNN setup as the base contract.
- Produces independent training runs for each grid point.

Current sweep grid:

- Learning rates: `5e-4`, `1e-3`, `2e-3`
- Noise factors: `0.0`, `0.03`, `0.05`

Primary use:

- Test whether noise injection improves EGNN long-horizon rollout behavior.

### Evaluate a Checkpoint

```bash
uv run python -m evaluation.evaluate \
  --config configs/egnn.yaml \
  --checkpoint checkpoints/egnn/<run_id>/best.pt \
  --test-path data/output/scaling/test.h5 \
  --output-dir results/evaluation/egnn/<run_id> \
  --device auto
```

Arguments:

| Argument | Required | Default | Description |
| --- | --- | --- | --- |
| `--config` | Yes | None | Model YAML used to rebuild the model. |
| `--checkpoint` | Yes | None | Path to the checkpoint to evaluate. |
| `--test-path` | No | `data/output/test.h5` | HDF5 test dataset. |
| `--output-dir` | No | `results/evaluation/<model>/<run_id>` | Directory for `metrics.json` and `summary.csv`. |
| `--device` | No | `auto` | Evaluation device. `auto` selects CUDA, then MPS, then CPU. |

What happens:

- Loads the model config and checkpoint.
- Reads the test trajectories.
- Runs one-step prediction over all test transitions.
- Runs autoregressive rollouts from each initial test state.
- Computes rollout curves, fixed-step summaries, divergence thresholds, and final finite-state fraction.
- Computes physical-energy drift from the predicted rollouts.
- Computes learned-Hamiltonian drift for HGNN checkpoints.
- Writes one complete JSON report and one flattened CSV summary.

Primary outputs:

- `metrics.json`: full evaluation report.
- `summary.csv`: flattened one-row summary for spreadsheet/table workflows.

### Build a Scaling Report

```bash
uv run python -m evaluation.scaling_report \
  --manifest runs/scaling_runs.yaml \
  --output runs/scaling_report.md
```

Arguments:

| Argument | Required | Default | Description |
| --- | --- | --- | --- |
| `--manifest` | Yes | None | YAML manifest pointing to evaluation reports by model and training size. |
| `--output` | No | None | Optional markdown file path. When omitted, the report is printed only. |

What happens:

- Loads evaluation reports listed in the manifest.
- Groups them by model and training-set size.
- Renders markdown tables for side-by-side comparison.
- Writes the markdown report if `--output` is provided.

Primary use:

- Produce thesis-ready tables after all official evaluations are complete.

## Colab Interface

The Colab entry point is:

```text
colab/train_colab.ipynb
```

The notebook is a launcher for one `(MODEL, N_TRAIN)` run at a time. It is designed for long GPU training with Google Drive persistence.

User parameters:

| Parameter | Values | Description |
| --- | --- | --- |
| `GITHUB_TOKEN` | GitHub token string | Used by Colab to clone the private or protected repository. |
| `MODEL` | `egnn`, `hgnn` | Selects the model configuration and output namespace. |
| `N_TRAIN` | `1000`, `2000`, `5000`, `10000` | Number of training trajectories to use from the Drive dataset. |
| `EPOCHS` | Integer | Number of epochs to run. |
| `RUN_TRAINING` | Boolean | Whether to run training. |
| `RUN_EVALUATION` | Boolean | Whether to evaluate the selected checkpoint after training. |

Expected Drive input:

```text
MyDrive/masters-thesis/data/scaling/train.h5
MyDrive/masters-thesis/data/scaling/val.h5
MyDrive/masters-thesis/data/scaling/test.h5
```

Drive output layout:

```text
MyDrive/masters-thesis/runs/<model>/n<N_TRAIN>/<run_id>/
```

The notebook validates that `train.h5` contains at least `N_TRAIN` trajectories before training. This prevents accidentally running a 2000-trajectory experiment against a 1000-trajectory training file.

## Configuration Contracts

### Data Generation Config

The data-generation config controls the physical setup and output paths.

| Field | Description |
| --- | --- |
| `train_path` | HDF5 path for generated training trajectories. |
| `val_path` | HDF5 path for generated validation trajectories. |
| `test_path` | HDF5 path for generated test trajectories. |
| `n_particles` | Number of particles per trajectory. Current thesis setup uses 3. |
| `n_train`, `n_val`, `n_test` | Number of trajectories per split. |
| `t_end` | Simulated time horizon. |
| `dt` | Time step between saved frames. |
| `G` | Gravitational constant used by the simulator. |
| `mass` | Particle mass used by the simulator. |
| `min_distance` | Rejection threshold for too-close initial states. |
| `max_position` | Position sampling scale. |
| `pos_scale` | Position normalization scale stored with the dataset. |
| `vel_scale` | Velocity normalization scale stored with the dataset. |
| `seed` | Random seed for reproducible generation. |

### Training Config

The training config is split into stable sections so experiments can be compared without changing code.

| Section | Description |
| --- | --- |
| `model` | Model family and architecture shape, including `name`, `hidden_dim`, and `n_layers`. |
| `data` | Training and validation HDF5 paths, time step, and optional training-trajectory cap. |
| `training` | Epoch count, batch size, learning rate, weight decay, loss, seed, device, and optional `noise_factor`. |
| `scheduler` | Learning-rate scheduler settings. |
| `checkpointing` | Checkpoint enablement and root directory. |
| `logging` | Metrics logging enablement and root directory. |

The canonical model names are `egnn` and `hgnn`. These names control model construction, output directory conventions, and report grouping.

## Artifact Contracts

### HDF5 Dataset

Each generated split is an HDF5 file with:

- `trajectories`: array shaped `(n_trajectories, n_frames, n_particles, 5)`.
- `energies`: physical-energy values for the simulated trajectories.
- metadata attributes for dataset shape, simulation parameters, normalization scales, random seed, and rejection rate.

The state channel order is:

```text
[x, y, vx, vy, mass]
```

The models learn transitions over `x`, `y`, `vx`, and `vy`; mass is retained as part of the particle state and physical context.

### Training Checkpoint

Each checkpoint stores:

- epoch number
- model state
- optimizer state
- validation loss
- full training config
- model name
- run id
- normalization statistics
- git commit when available

`best.pt` is the checkpoint selected by lowest validation loss. `latest.pt` is the most recent epoch checkpoint.

### Training Metrics CSV

Training writes a compact CSV with:

```text
epoch,train_loss,val_loss,lr
```

This file is useful for monitoring optimization, but it is not the official evaluation result.

### Evaluation JSON

`metrics.json` is the canonical evaluation artifact. Its top-level sections are:

| Section | Description |
| --- | --- |
| `metadata` | Model, checkpoint, config, test dataset, device, run id, normalization, and dataset shape. |
| `single_step` | One-step MSE and closest-pair distance summaries. |
| `rollout` | Autoregressive rollout summaries, full curves, divergence thresholds, and finite-state fraction. |
| `energy` | Physical-energy drift, plus learned-Hamiltonian drift for HGNN. |

The evaluator serializes invalid numeric values as `null` in JSON. This is intentional: divergence is evidence, and the report should expose it without breaking downstream readers.

### Evaluation Summary CSV

`summary.csv` is a flattened one-row view of the evaluation report. It exists for spreadsheet use and thesis-table drafting. The JSON file remains the authoritative artifact.

## Evaluation Semantics

The evaluator measures four classes of behavior.

Single-step prediction:

- Measures immediate transition accuracy across all adjacent test frames.
- Reports MSE statistics over the predicted state channels.

Rollout accuracy:

- Starts from the first frame of each test trajectory.
- Feeds each prediction back into the model to produce the next state.
- Reports fixed-step MSE summaries and full per-step curves.

Divergence behavior:

- Tracks non-finite predictions.
- Tracks first crossing steps for configured rollout-MSE thresholds.
- Reports the final fraction of trajectories that remain finite.

Energy behavior:

- Computes physical-energy drift over predicted rollouts.
- For HGNN, also reports drift of the learned Hamiltonian quantity.
- Uses energy as a stability and structure diagnostic, not as a replacement for rollout error.

## Validation

The product is validated by unit tests around the code contracts that matter for repeatable experiments:

- data generation and HDF5 I/O
- dataset transition loading
- training I/O and checkpoint contracts
- evaluation metric computation
- evaluation report serialization
- scaling-report loading and rendering

Run the full local validation suite with:

```bash
uv run pytest
```

Run formatting and lint checks with:

```bash
uv run ruff format --check .
uv run ruff check .
```

Validation complete means the code passes the test suite and an official evaluation can be generated from a trained checkpoint. For thesis results, passing tests is necessary but not sufficient: the corresponding `metrics.json` files must also be preserved.

## Operating Model

The workbench is intended to be operated as a controlled experiment pipeline.

Recommended operating rules:

- Keep dataset files stable during a comparison.
- Compare EGNN and HGNN under matching train, validation, and test splits.
- Treat `best.pt` as the default checkpoint for official evaluation unless a specific reason is documented.
- Keep each evaluation output directory tied to one checkpoint.
- Preserve raw `metrics.json` files before writing interpretation.
- Use scaling reports to generate comparison tables rather than manually transcribing values from notebooks.
- Use notebooks for visual diagnosis after the numeric evidence is in place.

For Colab runs:

- Store data and run outputs in Google Drive.
- Run one `(MODEL, N_TRAIN)` pair per notebook execution.
- Confirm the output directory contains both checkpoint files and evaluation files before deleting or moving a run.

## Known Boundaries

- The current experimental setup focuses on 3-particle gravitational dynamics.
- The current official models are EGNN and HGNN.
- The training objective is one-step supervised MSE; long-horizon behavior is measured during evaluation.
- The evaluator assumes the test file follows the repository HDF5 dataset contract.
- Scaling reports depend on a manifest that points to existing `metrics.json` files.
- Colab reliability depends on the selected runtime, Google account limits, and Drive availability.

These boundaries are deliberate. They keep the project narrow enough that thesis claims can be tied to controlled evidence rather than a broad collection of loosely related experiments.

## Q&A

Why is `metrics.json` the official result instead of plots?

Plots are useful for inspection, but they are not stable enough for thesis tables. `metrics.json` records the full numeric report, including rollout curves and divergence information, in a machine-readable format.

Why does evaluation include both rollout error and energy drift?

Rollout error answers whether the predicted trajectory stays close to the simulated trajectory. Energy drift answers whether the predicted dynamics preserve a key physical invariant. They measure different failure modes, so both are needed.

Why evaluate autoregressively if the models train one step at a time?

The thesis question is about learned dynamics over time. One-step accuracy is necessary, but a model can look good one step ahead and still drift or diverge when its own predictions become the next input.

Why keep `--n-train` as a command-line override?

It makes scaling experiments explicit at invocation time. The same base config can be reused for 1000, 2000, and 5000 trajectory runs without producing near-duplicate YAML files.

Why does Colab validate the number of trajectories in `train.h5`?

The most expensive failure is running a long experiment on the wrong dataset. The notebook checks the file before training so a 2000 or 5000 trajectory run cannot silently use a smaller training file.

When should a result be considered ready for thesis discussion?

A result is ready when the checkpoint is preserved, evaluation has completed, `metrics.json` and `summary.csv` are available, and the scaling report or thesis table can be regenerated from those files.
